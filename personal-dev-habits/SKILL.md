# Personal Dev Habits

## 使用原则

将这些规则作为跨项目的个人偏好补充。项目已有规范、业务 skill、框架约束和用户明确要求优先；没有冲突时，按本 skill 保持代码更容易阅读、排查和维护。

## 代码注释

关键函数、关键变量、关键业务分支需要补充简洁的中文注释。注释重点说明业务含义、边界条件、数据来源、状态流转原因，避免重复描述代码本身。

适合添加注释的场景：
- 函数承载核心业务流程、状态变更、事务边界、外部接口调用、复杂 SQL 参数组装。
- 变量代表业务状态、配置开关、特殊枚举、跨系统字段、容易误解的中间结果。
- 判断条件来自历史兼容、产品规则、数据修复、异常兜底等上下文，单看代码不容易理解。

不需要为了普通 getter/setter、简单赋值、明显的循环和临时局部变量添加注释。注释应短、准、贴近代码位置，优先使用中文。

## 函数封装

保持逻辑就近可读，减少不必要的函数拆分。只有一个地方使用、逻辑很短、命名不能显著提升理解的代码，优先留在原函数中。

适合封装的场景：
- 同一段逻辑会被多个地方复用。
- 原函数已经过长，拆分后能形成清晰的业务步骤。
- 逻辑有独立的输入输出、异常处理或测试价值。
- 封装后的函数名能准确表达业务意图，减少阅读成本。

避免为了“看起来更整洁”把代码拆成过多私有函数。每次新增 helper 前先问：这个函数是否真实降低复杂度，还是让读者不得不来回跳转。

## Java 后端事务

修改 Java 后端代码前，先理清事务如何生效，再决定改动位置和调用方式。

排查顺序：
1. 优先查看目标类、方法、调用链上是否存在 `@Transactional`，确认事务传播行为、回滚异常类型和只读配置。
2. 如果事务通过 Spring AOP/XML/配置类统一配置，找到对应切面配置，确认 pointcut 路径、类名或方法名匹配规则。
3. 注意 AOP 事务常受包路径、方法命名、代理调用方式影响；新增或改名方法时，确认仍能命中事务切面。
4. 避免在同类内部自调用事务方法后误以为代理会生效；如果已有项目采用代理暴露、Service 拆分或编程式事务，沿用现有模式。
5. 涉及多次写库、外部接口、消息发送、异步任务时，明确哪些操作在事务内，哪些操作应在提交后执行或需要补偿。

交付前说明事务判断结果，尤其是新增方法是否命中 AOP 规则、回滚条件是否符合业务预期。
